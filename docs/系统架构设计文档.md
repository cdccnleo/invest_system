## 2 系统概述

### 2.1 系统定位

InvestPilot 是一套面向个人投资者的**半自动化投资分析系统**。系统每日自动完成数据采集、持仓分析、风险监控与决策支持，并在关键节点通过主动对话推送投资洞察。系统当前处于 **L3 投资伙伴** 成熟度等级（Hermes Agent Co-evolution 模型），具备 5 项核心主动能力：主动对话引擎、行为画像分析、压力测试模拟、情绪感知监测、技能固化联动。

### 2.2 核心目标

| 目标 | 说明 |
|------|------|
| 自动化数据采集 | 每日盘后自动获取持仓股行情、财务、公告、研报、新闻数据 |
| 结构化分析记忆 | 为每只持仓股维护完整的 8 章分析记忆文件（TAMF） |
| 主动风险预警 | 基于多维度信号（偏离度、新闻情感、评级变动）生成风险预警 |
| 智能决策支持 | 集成 DeepSeek + Ollama 双引擎，提供投资分析与建议 |
| 可进化学习 | 通过技能固化机制，将重复分析模式沉淀为可复用的技能 |

### 2.3 系统边界

| 范围 | 包含 | 不包含 |
|------|------|--------|
| 数据范围 | 沪深 A 股（含 ETF） | 港股、美股、期货、外汇 |
| 数据来源 | 公开行情 + 公告 + 研报 | 付费数据源、券商接口 |
| 分析深度 | 基本面 + 技术面 + 消息面 | 量化回测（数据积累中） |
| 交易执行 | 仅分析建议 | 自动交易、券商直连（Phase 5 规划） |
| 部署方式 | 单机部署（WSL2 / Linux） | 集群部署、微服务化 |

### 3.2 架构层次说明

| 层次 | 职责 | 关键技术决策 |
|------|------|-------------|
| **交互表现层** | 用户交互界面、数据可视化、推送通知 | Streamlit 单页应用，多 Tab 模块化架构 |
| **调度层** | 定时任务编排、工作流编排 | APScheduler 独立调度，不与系统 cron 冲突 |
| **分析引擎层** | 核心分析逻辑、趋势预测、回测评估 | 独立模块化设计，通过函数调用组合分析管线 |
| **Prompt组装层** | 多源数据融合、LLM 提示词构建 | TAMF 摘要注入策略，预估节省 30%+ Token |
| **LLM集成层** | 多模型路由、API 调用封装、容错 | DeepSeek + Ollama 双引擎，模型路由按任务类型选择 |
| **数据采集层** | 多数据源采集、清洗、归一化 | OpenClaw + 自研采集器，单标的数据轮询 |
| **数据持久化层** | 结构化存储、向量检索、文件备份 | PostgreSQL 为主 + 文件系统降级 |

### 3.3 架构设计原则

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个模块只负责一个功能域，模块间通过函数调用解耦 |
| **数据优先** | 所有分析的最终输出写回数据库或文件系统，确保可追溯 |
| **降级优先** | 每个外部依赖（DB、LLM API）都有降级路径，单点故障不阻塞全系统 |
| **影子安全** | TAMF 文件更新采用影子模式，验证通过后才晋升到生产目录 |
| **可审计** | 所有关键操作写入时间线事件表和审计日志 |

### 4.2 TAMF 投资记忆系统

#### 4.2.1 概述

TAMF（Target Analysis Memory File）是 InvestPilot 的核心创新——为每只持仓标的维护一份结构化的 8 章分析记忆文件。系统当前为 **46 只持仓标的** 生成了完整的 TAMF 文件。

#### 4.2.2 TAMF 文件结构（8 章）

| 章节 | 名称 | 内容来源 | 更新频率 |
|:---:|------|---------|:-------:|
| 一 | 基本档案 | 行业、市值、主营业务 | 首次生成 + 财报触发 |
| 二 | 持仓历程 | 开仓日期、操作历史、成本、盈亏 | 交易执行事件驱动 |
| 三 | 基本面分析 | 营收/利润趋势、ROE、估值分位 | Agent 每季度生成 |
| 四 | 技术面分析 | 均线、MACD、RSI、支撑/阻力 | Agent 每周生成 |
| 五 | 消息面摘要 | 新闻情感趋势、热度统计 | 每日增量更新 |
| 六 | 相关标的监控 | 嵌入向量相似度关联的标的 | 每日更新（待激活） |
| 七 | 监控与预警 | 风险评分、评级变动、偏离度 | 事件驱动 |
| 八 | 反思与笔记 | Agent 自我评估、计划修正 | 深度分析触发 |

#### 4.2.3 更新模式

TAMF 系统支持三种更新模式：

| 模式 | 触发方式 | 安全机制 |
|------|---------|---------|
| **定时增量更新** | APScheduler 每日 15:35 | 影子模式隔离写入，差异对比后晋升 |
| **事件驱动更新** | 交易执行 / 公告发布 / 评级变动 | 直接写入特定章节，不影响其他章节 |
| **周频深度分析** | APScheduler 周日 22:00 | Agent 生成完整趋势预测段落 |

#### 4.2.4 手动编辑保护

通过 `<!-- MANUAL EDIT -->` 标记机制保护用户手动编辑的段落不被自动更新覆盖：

```
检测流程:
  写入新内容前 → is_section_manually_edited() 正则扫描
    → 若含 MANUAL EDIT 标记 → 跳过该段落
    → 若不含 → 正常覆盖更新
```

#### 4.2.5 影子模式

| 组件 | 文件 | 功能 |
|------|------|------|
| TamfShadowMode 类 | `tamf_shadow.py` | 影子写入、差异对比（difflib）、晋升/回滚 |
| CLI 管理 | `schedule_runner.py --mode tamf-shadow` | enable/disable/status/diff/promote-all |

工作流：
```
增量更新 → 影子目录 (target_memories_shadow/)
  → difflib.unified_diff 对比
  → 推送差异通知
  → 人工/自动 promote 到生产目录
  → promote 前自动备份生产文件
```

### 4.3 L3 投资伙伴系统

#### 4.3.1 主动对话引擎

| 组件 | 文件 | 说明 |
|------|------|------|
| L3DialogEngine | `l3_dialog_engine.py` | 核心引擎，管理 5 种触发器和对话状态 |
| 行为画像 | `audit_analytics.py` | 基于 ≥30 天审计日志的行为基线 |
| 压力测试 | `backtest_engine.py` | 5 种极端情景（黑色星期一/利率飙升/行业黑天鹅/流动性危机/股债双杀） |
| 情绪感知 | `l3_dialog_engine.py` | 基于新闻情感、市场情绪的短期偏离检测 |

#### 4.3.2 触发器类型

| 触发器 | 触发条件 | 推送方式 | 优先级 |
|--------|---------|---------|:---:|
| `deviation_alert` | 偏离度超过阈值 | 盘中即时推送 | 🔴 高 |
| `periodic_checkin` | 盘前 07:30 / 晚间 20:00 | 定时推送 | 🟡 中 |

## 7 技术栈选型与理由

### 7.1 技术栈总表

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|:---:|---------|
| **编程语言** | Python | 3.11 | 生态丰富（金融/ML/NLP），与 Hermes Agent 兼容 |
| **Web 框架** | Streamlit | 1.28+ | 零前端经验可快速构建数据仪表盘，热重载开发效率高 |
| **任务调度** | APScheduler | 3.10+ | 纯 Python 进程内调度，无外部依赖，支持 Cron 表达式 |
| **主数据库** | PostgreSQL | 16 | 金融数据关系模型天然适配，支持 pgvector 扩展 |
| **向量检索** | pgvector + HNSW | 0.7+ | 与主数据库统一管理，无需额外部署向量数据库 |
| **列级加密** | pgcrypto | PG 内置 | 无需额外依赖，SQL 级别透明加密 |
| **LLM 主引擎** | DeepSeek API | — | 中文金融问答质量经实测验证优于 GPT-4，API 成本低 |
| **LLM 本地引擎** | Ollama | 0.3+ | 本地部署敏感分析任务，无数据外泄风险，无 API 费用 |
| **数据采集** | OpenClaw + requests | — | OpenClaw 窗口管理器解决反爬，requests 轻量高效 |
| **通知通道** | requests (Webhook) | — | 微信/飞书/Bark 均基于 HTTP Webhook，统一调用接口 |
| **加密工具** | cryptography | 41+ | Python 生态标准加密库，Fernet 对称加密 + RSA 非对称 |
| **数据清洗** | pandas + numpy | 2.0+ | 金融时间序列分析行业标准 |

### 7.2 核心选型决策记录

| 决策 | 选项 | 选择 | 理由 |
|------|------|:---:|------|
| LLM 架构 | 纯 API / 纯本地 / **混合** | 混合 | DeepSeek 做复杂推理，Ollama 做本地敏感分析，兼顾质量与隐私 |
| 任务调度 | cron / APScheduler / Celery | APScheduler | 纯 Python 零部署成本，Celery 对单机应用过重 |
| 向量数据库 | pgvector / Milvus / Pinecone | pgvector | 与业务数据同库，无需额外运维，HNSW 索引性能足够 |
| 数据持久化 | PostgreSQL / 文件系统 / **混合** | 混合 | PostgreSQL 为主存储，TAMF .md 文件作为人类可读的分析记忆 |
| 仪表盘 | Streamlit / Dash / Gradio | Streamlit | 快速原型能力最强，单文件即可构建完整仪表盘 |
| 推送通知 | 微信 / 飞书 / **多渠道** | 多渠道 | 不依赖单一平台，Server酱/飞书/Bark 同时推送 |

## 10 安全策略

### 10.1 安全架构（5 层体系）

```
Layer 1: 列级加密 (pgcrypto)
  ├─ credentials 表中的敏感数据加密存储
  └─ pgp_sym_encrypt / pgp_sym_decrypt 函数

Layer 2: 传输脱敏 (data_sanitizer.py)
  ├─ 云端数据在进入 LLM API 前清洗
  └─ 正则匹配脱敏规则

Layer 3: 凭据隔离 (credentials.py)
  ├─ 凭据文件 ~/.hermes/invest_credentials/（已 gitignored）
  ├─ 多层级读取（WCM → 文件 → 环境变量）
  └─ 无明文密码硬编码

Layer 4: 加密传输 (encryption_helper.py)
  ├─ Fernet 对称加密（本地缓存/配置文件）
  └─ RSA 非对称加密（跨机器传输）

Layer 5: 审计追踪 (audit_log)
  ├─ 所有 LLM 调用记录
  ├─ 所有关键数据更新记录
  └─ 所有异常操作记录
```

| 扩展类型 | 操作方式 | 示例 |

| 新增 LLM 模型 | 修改 `model_router.py` + `llm_caller.py` | 如接入 GPT-4o |

### 12.2 数据源扩展

新数据源接入标准流程：

```
1. 创建 scripts/fetch_<source>.py
   ├─ 实现采集函数
   └─ 写入对应的 DB 表

2. 注册到 schedule_runner.py
   ├─ 添加采集定时任务
   └─ 添加数据后处理钩子

3. 集成到 TAMF 更新
   ├─ tamf_updater.detect_new_data_for_target() 增加检测
   └─ tamf_updater.incremental_update() 增加更新逻辑

4. 集成到 prompt_builder
   └─ 在 build_tamf_summaries_for_prompt() 中注入新数据源摘要
```
