## 3 技术实现可行性

### 3.1 实现方案设计

#### 方案一：单按钮全局同步（推荐）

在每个页面顶部放置一个"🔄 手动同步"按钮，点击后触发对应数据类型的采集：

```
┌─────────────────────────────────────────────┐
│  📰 新闻摘要（近7日）          [🔄 同步新闻]  │
│  最后同步: 2026-05-31 21:00   ⏳ 同步中...  │
├─────────────────────────────────────────────┤
│  [新闻列表...]                               │
└─────────────────────────────────────────────┘
```

#### 方案二：分类独立同步按钮

在三个页面分别放置独立的同步按钮，新闻/公告/研报各触发各自的采集函数。

**推荐方案一**，理由：
- 实现简单，改动量最小
- 用户操作直观
- 与现有"持仓同步"按钮模式一致

**总改动量**：约 105 行代码，3 个文件

### 3.3 核心实现伪代码

```python
# dashboard_views/_news.py 中的改造

def render_news_summary():
    st.markdown("## 📰 新闻摘要（近7日）")

    # ── 手动同步按钮 ──
    col_title, col_btn = st.columns([3, 1])
    with col_btn:
        if st.button("🔄 同步新闻", help="手动触发新闻数据采集", key="sync_news_btn"):
            with st.spinner("正在采集新闻数据..."):
                try:
                    from fetch_news import collect_news
                    from fetch_news import save_news_to_db  # 新增函数
                    news = collect_news()
                    saved = save_news_to_db(news)
                    st.session_state["news_last_sync"] = datetime.now()
                    st.success(f"同步完成：采集 {len(news)} 条，新增 {saved} 条")
                    st.rerun()
                except Exception as e:
                    st.error(f"同步失败: {e}")

    # 显示上次同步时间
    with col_title:
        last_sync = st.session_state.get("news_last_sync")
        if last_sync:
            st.caption(f"最后同步: {last_sync.strftime('%m-%d %H:%M:%S')}")

    # ... 原有展示逻辑 ...
```

### 3.4 关键技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 同步方式 | 同步阻塞（非异步） | Streamlit 本身是同步模型；采集耗时 < 30s 可接受 |
| 按钮放置 | 页面顶部 | 与现有"持仓同步"按钮一致，用户预期统一 |
| 执行反馈 | `st.spinner` + `st.toast/success` | Streamlit 原生组件，无需额外依赖 |
| 去重策略 | 复用现有 `ON CONFLICT DO NOTHING` | 避免重复数据，与定时任务一致 |
| 数据刷新 | `st.rerun()` | 同步完成后强制刷新页面展示最新数据 |

## 4 系统性能影响评估

### 4.1 单次同步耗时估算

基于现有采集函数在本地环境的实测数据估算：

| 数据类型 | 采集函数 | 网络请求数 | 预估耗时 | 瓶颈 |
|---------|---------|:---:|:---:|------|
| 新闻 | `collect_news()` | 4-5 个外部 API | 5-15 秒 | 财联社网页版需解析 HTML |
| 研报 | `collect_reports(days_back=7)` | 10 页 × 单次请求 | 10-20 秒 | 东方财富 API 分页 |
| 公告 | `fetch_all_positions_announcements()` | N 只股票 × 3 页 | 15-60 秒 | 持仓数量决定（46只 × 请求间隔0.3s） |

**公告同步是最耗时的场景**，按当前 46 只持仓计算：
- 每只股票 3 页请求，间隔 0.3 秒
- 估算：46 × 3 × (0.3s + 请求耗时~0.5s) ≈ **110 秒**

### 4.2 服务器负载分析

| 资源 | 手动同步影响 | 与定时任务对比 |
|------|------------|--------------|
| CPU | 低（主要是 HTTP 等待，非计算密集型） | 相同 |
| 内存 | 极低（数据在内存中临时存储即写入DB） | 相同 |
| 网络 | 中（并发请求外部 API） | 相同 |
| 数据库 | 低（INSERT ON CONFLICT 操作） | 相同 |

**结论**：手动同步与定时任务执行相同的采集逻辑，对服务器负载**无明显额外影响**。

### 4.3 优化建议

| 优化项 | 预期效果 | 实施难度 |
|--------|---------|:---:|
| 公告采集添加进度条 | 改善长时间等待体验 | 低 |
| 新闻采集使用异步并发 | 将 4 源串行 → 并行，耗时减少 50%+ | 中（已有 `async_collector.py`） |
| 添加"仅同步今日"选项 | 公告采集减少到 1 天窗口，大幅缩短耗时 | 低 |
| 后台线程执行 | 避免阻塞 Streamlit UI | 高（需改造 Streamlit 渲染模型） |

## 6 数据安全与一致性

### 6.1 数据一致性保障

| 机制 | 实现方式 | 保障效果 |
|------|---------|---------|
| 数据库去重 | `INSERT ... ON CONFLICT DO NOTHING` | 防止重复数据 |
| 采集函数去重 | `seen_titles` 集合（新闻）+ `seen_ids` 集合（公告/研报） | 同次采集内去重 |
| 事务性写入 | `conn.commit()` 整体提交 | 部分失败不影响已有数据 |
| 异常回滚 | `conn.rollback()` on error | 写入失败不产生脏数据 |

### 6.2 并发安全分析

| 场景 | 风险评估 | 应对措施 |
|------|:---:|------|
| 手动同步 + 定时任务同时运行 | 🟡 低风险 | APScheduler 已配置 `max_instances=1`，同任务不会并发；不同任务（如新闻 vs 研报）操作不同表，无冲突 |
| 两个用户同时点击同步 | 🟢 无风险 | Streamlit 单用户 Session，每次只有一个连接 |
| 同步过程中数据库连接断开 | 🟡 低风险 | 采集函数已有 `try/except` 保护，`finally` 中关闭连接 |

### 6.3 异常处理机制

```python
def safe_sync_news():
    """安全同步新闻，所有异常均有降级处理"""
    try:
        news = collect_news()
        if not news:
            return {"status": "empty", "total": 0, "new": 0}
        saved = save_news_to_db(news)
        return {"status": "ok", "total": len(news), "new": saved}
    except requests.Timeout:
        return {"status": "timeout", "total": 0, "new": 0}
    except Exception as e:
        logger.error(f"新闻同步失败: {e}")
        return {"status": "error", "total": 0, "new": 0, "error": str(e)}
```

## 7 与现有自动同步机制的兼容性

### 7.1 定时任务与手动同步的关系

```
                定时任务（自动）              手动同步（按需）
                ──────────────              ──────────────
触发方式:       Cron 表达式定时             用户点击按钮
触发频率:       每日 1 次（固定时间）        任意时刻
数据范围:       固定窗口（7天/30天）         可配置窗口
执行环境:       后台调度器线程               Streamlit 主线程
冲突处理:       coalesce=True 合并         button disabled 禁用
```

### 7.2 优先级与冲突处理

| 场景 | 处理策略 |
|------|---------|
| 定时任务正在运行，用户点击同步 | 按钮显示"定时同步进行中，请稍候..."，不发起新请求 |
| 用户手动同步刚完成，定时任务触发 | 定时任务正常执行，`ON CONFLICT DO NOTHING` 保证不重复写入 |
| 同一用户快速双击同步按钮 | `st.button` 天然防抖 + `disabled` 状态控制 |

### 7.3 同步状态共享

建议在 `session_state` 中维护一个全局同步状态字典：

```python
# dashboard_views/_shared.py
if "sync_status" not in st.session_state:
    st.session_state["sync_status"] = {
        "news": {"last_sync": None, "syncing": False},
        "reports": {"last_sync": None, "syncing": False},
        "announcements": {"last_sync": None, "syncing": False},
    }
```

### 7.4 与现有机制的兼容性总结

| 兼容性维度 | 评估 | 说明 |
|-----------|:---:|------|
| 数据库 Schema | ✅ 完全兼容 | 复用现有表结构，无需变更 |
| 去重机制 | ✅ 完全兼容 | 复用 `ON CONFLICT DO NOTHING` |
| 定时任务 | ✅ 无冲突 | 不同触发方式，数据库层面去重 |
| Streamlit 渲染 | ✅ 兼容 | 使用标准 Streamlit 组件 |
| 审计日志 | ✅ 可扩展 | 可选添加 `MANUAL_SYNC_*` 审计事件 |

## 9 潜在风险与应对策略

### 9.1 风险登记册

| 风险ID | 风险描述 | 概率 | 影响 | 应对策略 |
|:---:|------|:---:|:---:|------|
| R1 | 公告同步耗时过长（>60s），用户等待不耐烦 | 高 | 中 | 添加进度条；提供"仅今日"选项；超时 120s 后自动提示 |
| R2 | 外部 API 不可用导致同步失败 | 中 | 低 | 显示明确错误信息；保留上次成功数据；各源独立 try/except |
| R3 | Streamlit 同步阻塞导致 UI 无响应 | 高 | 低 | 使用 `st.spinner` 提供视觉反馈；`st.button` 天然防重复点击 |
| R4 | 手动同步与定时任务产生重复数据 | 低 | 低 | `ON CONFLICT DO NOTHING` 自动去重 |
| R5 | 新闻采集函数无 DB 写入，需新增代码 | 确定 | 低 | 补充 `save_news_to_db()` 函数，约 30 行代码 |

### 9.2 应急预案

| 场景 | 应急措施 |
|------|---------|
| 同步超时 | 设置 120 秒超时，超时后显示"同步超时，请稍后重试" |
| API 全部不可用 | 显示"所有数据源暂不可用，请检查网络连接" |
| DB 写入失败 | 保持已有数据不变，显示"数据保存失败，请检查数据库连接" |
| 按钮被误触 | 提供 3 秒冷却期（`time.sleep(3)` 后按钮才可再次点击） |

### 11.2 实施建议

1. **分阶段实施**：优先实现新闻同步按钮（改动最小、用户感知最强），再逐步添加研报和公告同步
2. **公告同步优化**：公告采集建议默认使用"仅今日"模式（1天窗口），完整模式（30天）作为高级选项
3. **审计日志**：建议为手动同步添加 `MANUAL_SYNC_NEWS` / `MANUAL_SYNC_REPORTS` / `MANUAL_SYNC_ANNOUNCEMENTS` 审计事件
4. **后续扩展**：可考虑在侧边栏添加"一键全量同步"按钮，同时触发新闻+公告+研报采集

### 11.3 实施优先级

| 优先级 | 任务 | 理由 |
|:---:|------|------|
| P0 | 新闻采集 DB 写入 + 同步按钮 | 新闻采集目前无 DB 写入，需优先补齐 |
| P1 | 研报同步按钮 | 采集函数已完备，直接复用 |
| P2 | 公告同步按钮 | 耗时较长，需添加进度优化 |
| P3 | 侧边栏全局同步状态 + 一键全量同步 | 锦上添花，提升整体体验 |

---

> **评估结论**：在新闻摘要、公告及研报仪表盘系统中增加手动同步按钮**技术上完全可行**，实施难度低（约 2.5-3 人天），与现有系统架构高度兼容，对性能和安全无明显负面影响。建议按 P0→P1→P2→P3 优先级分阶段实施。
