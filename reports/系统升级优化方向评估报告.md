## 6 性能优化

### 6.1 P1 — 并行处理激活（P1）

#### 现状分析
系统架构文档已明确标注 TAMF 并行更新为"待激活"状态。当前 [tamf_updater.py](file:///c:/PythonProject/invest_system/scripts/tamf_updater.py) 遍历 46 只标的时采用单线程顺序处理，每只标的涉及 5-6 次数据库查询，总耗时约 5 分钟。

#### 技术方案

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def parallel_update_all(holdings: list[str], max_workers: int = 4) -> dict:
    """
    并行更新所有持仓标的的 TAMF 文件。
    max_workers=4 可避免数据库连接池耗尽和 API 限流。
    """
    results = {"success": 0, "failed": 0, "skipped": 0, "details": {}}

    with ThreadPoolExecutor(max_workers=max_workers) as executor:
        futures = {
            executor.submit(incremental_update, code): code
            for code in holdings
        }
        for future in as_completed(futures):
            code = futures[future]
            try:
                result = future.result(timeout=120)
                results["details"][code] = result
                if result.get("updated"):
                    results["success"] += 1
                else:
                    results["skipped"] += 1
            except Exception as e:
                results["failed"] += 1
                results["details"][code] = {"error": str(e)}
                logger.error(f"TAMF 并行更新失败: {code}, {e}")

    return results
```

同时将数据采集层的 6 个采集器也改为并行执行（盘后 15:00 同时采集行情、财务、公司行为）。

| 技术可行性 | 5/5 | Python ThreadPoolExecutor 标准库，无额外依赖 |
| 实施复杂度 | 2/5 | 代码改动量小，主要是包裹现有函数 |
| 预期效益 | 4/5 | TAMF 更新耗时预计从 5 分钟降至 1.5 分钟 |

### 6.2 P2 — LLM 调用性能优化（P1）

#### 现状分析

当前 LLM 调用存在以下问题：
- 重复调用：相同或相似的 prompt 每次重新请求 LLM
- 无响应缓存：盘前简报内容可能多日相似
- 无批处理：多个标的的分析请求串行发送

#### 技术方案

```
1. 语义缓存（Semantic Cache）
   - 对相似的 prompt 计算 embedding 相似度
   - 相似度 > 0.95 时直接返回缓存结果
   - 缓存 TTL: 24 小时（行情数据日级变化）

2. 批处理优化
   - 将 46 只标的的分析请求合并为 1 次 LLM 调用
   - 在 system prompt 中指定输出 JSON 数组格式
   - 预计减少 90% 的 API 调用次数

3. Token 压缩增强
   - context_compressor.py 已有基础实现
   - 增强：TAMF 摘要生成时去除冗余信息
   - 增强：使用更激进的截断策略
```

#### 可行性评估

| 维度 | 评分 | 说明 |
|------|:---:|------|
| 技术可行性 | 4/5 | 语义缓存需要 embedding 计算，但技术路径清晰 |
| 实施复杂度 | 3/5 | 批处理需要重构 prompt 构建逻辑 |

## 8 数据安全强化

### 8.1 S1 — 数据库自动备份与恢复（P1）

#### 现状分析
系统当前**无自动数据库备份机制**。唯一的数据保护是 TAMF 文件的 Git 版本控制和影子模式。PostgreSQL 数据库中的行情数据、财务数据、审计日志等一旦丢失将无法恢复。

#### 技术方案

```python
# backup_manager.py
class DatabaseBackupManager:
    """
    数据库自动备份管理器。
    策略:
      - 每日增量备份: WAL 归档（Write-Ahead Log）
      - 每周全量备份: pg_dump 自定义格式
      - 保留策略: 增量保留 7 天，全量保留 4 周
    """

    def full_backup(self) -> str:
        """执行全量备份，返回备份文件路径"""
        ...

    def incremental_backup(self) -> str:
        """执行 WAL 增量备份"""
        ...

    def restore(self, backup_path: str) -> bool:
        """从备份恢复数据库"""
        ...

    def verify_backup(self, backup_path: str) -> bool:
        """验证备份文件完整性"""
        ...
```

```
备份策略:
  全量备份: 每周日 23:00（APScheduler 定时任务）
  增量备份: 每日 02:00（WAL 归档）
  备份存储: data/backups/ 目录（建议同步到云存储）
  保留策略: 全量 4 周 + 增量 7 天
```

#### 可行性评估

| 维度 | 评分 | 说明 |
|------|:---:|------|
| 技术可行性 | 5/5 | pg_dump + WAL 归档是 PostgreSQL 标准功能 |
| 实施复杂度 | 2/5 | 主要是配置和脚本化 |
| 预期效益 | 4/5 | 消除数据丢失风险 |
| 潜在风险 | 1/5 | 低风险 |

**总周期**: 3 天 | **资源**: 1 人
